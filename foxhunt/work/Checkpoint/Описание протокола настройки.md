# Протокол настройки контрольного пункта FoxAss

Документ для разработчика веб-интерфейса конфигурирования КП.

---

## 1. Сетевой уровень

| Параметр | Значение |
|----------|----------|
| Тип сети | WiFi 802.11 b/g/n, точка доступа (AP) |
| SSID | `FoxAss-CP-FF` до конфигурирования, `FoxAss-CP-{cp_num_hex}` после (например `FoxAss-CP-05`) |
| Пароль | нет (открытая сеть) |
| IP адрес КП | `192.168.4.1` |
| Порт | `80` |
| Протокол | HTTP/1.1 |
| Канал | 6 |

**SSID после настройки** — `cp_num` представлен как два шестнадцатеричных символа без `0x`: КП №1 → `FoxAss-CP-01`, финиш (254) → `FoxAss-CP-FE`.

---

## 2. Общие соглашения

- Все запросы и ответы — `Content-Type: application/json`, кодировка UTF-8
- Числа — целые, без кавычек
- Булевы — `true` / `false` (не `1` / `0`)
- `master_secret` — строка из 32 шестнадцатеричных символов нижнего регистра (16 байт)
- Все поля в POST `/api/config` опциональны: передаются только изменяемые поля; остальные остаются прежними

**Коды HTTP ответов:**

| Код | Смысл |
|-----|-------|
| 200 | Успех |
| 400 | Неверный запрос (отсутствующие обязательные поля, неверный формат) |
| 500 | Внутренняя ошибка КП (например, ошибка записи NVS) |

---

## 3. Поля конфигурации

| Поле | Тип | Описание |
|------|-----|----------|
| `cp_num` | uint8 (0–254) | Номер КП. `0` = старт, `254` (0xFE) = финиш, `1–253` = обычный КП. Влияет на SSID |
| `event_id` | uint16 | ID соревнования. Используется в LoRa протоколе |
| `station_key` | string (32 hex-символа) | Ключ этого КП для MAC-подписи отметок на карте. **Уникален для каждого КП.** |
| `sector_key` | string (12 hex-символов) | 6-байтный ключ для аутентификации MIFARE-секторов. **Общий для всех КП одного соревнования.** |
| `station_type` | string (readonly) | Вычисляемое поле: `"start"` / `"finish"` / `"fox"` / `"regular"`. Только в ответе GET, не принимается в POST |
| `is_fox` | bool | Режим лиса (Fox-O): КП работает как обычный КП (пишет карты), но дополнительно периодически передаёт позывной в эфир по FSK |
| `fox_on_sec` | uint32 | Длительность передачи сигнала лиса, секунды (по умолчанию 15) |
| `fox_off_sec` | uint32 | Длительность паузы между передачами, секунды (по умолчанию 60) |
| `fox_morse` | string (макс 15 символов) | Позывной в морзянке (например `"MO1"`, `"MO2"`, `"MO5"`) |
| `has_task` | bool | Включить режим двойной записи с дополнительным заданием |
| `task_type` | uint8 | Тип задания: `0` = нет, `1` = алко-тест (MQ-3) |
| `task_bonus` | uint16 | Зарезервировано для системы очков (сейчас не используется) |
| `lora_freq_hz` | uint32 | Рабочая частота LoRa в Гц (по умолчанию 433 175 000) |
| `fox_freq_hz` | uint32 | Частота передатчика лиса в Гц (по умолчанию 433 920 000) |
| `initialized` | bool (readonly) | `true` если конфиг был записан хотя бы раз. Только в ответе GET |

---

## 4. Эндпоинты

### `GET /`

Статическая HTML-страница с описанием API (placeholder). Используется для проверки доступности сервера.

---

### `GET /api/config`

Получить текущую конфигурацию КП.

**Ответ 200:**
```json
{
  "cp_num": 5,
  "event_id": 2025,
  "station_key": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6",
  "sector_key": "1a2b3c4d5e6f",
  "station_type": "regular",
  "is_fox": false,
  "fox_on_sec": 15,
  "fox_off_sec": 60,
  "fox_morse": "MO1",
  "has_task": false,
  "task_type": 0,
  "task_bonus": 0,
  "lora_freq_hz": 433175000,
  "fox_freq_hz": 433920000,
  "initialized": true
}
```

**Значения `station_type`:**

| Значение | Условие |
|----------|---------|
| `"start"` | `cp_num == 0` |
| `"finish"` | `cp_num == 254` |
| `"fox"` | `is_fox == true` (и `cp_num != 0, 254`) — КП пишет карты и передаёт FSK морзянку |
| `"regular"` | остальные случаи |

---

### `POST /api/config`

Записать конфигурацию. Все поля опциональны — передаются только изменяемые. После успешной записи конфиг сохраняется в NVS и переживает перезагрузку.

**Тело запроса (все поля опциональны):**
```json
{
  "cp_num": 5,
  "event_id": 2025,
  "station_key": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6",
  "sector_key": "1a2b3c4d5e6f",
  "is_fox": false,
  "fox_on_sec": 15,
  "fox_off_sec": 60,
  "fox_morse": "MO1",
  "has_task": false,
  "task_type": 0,
  "task_bonus": 0,
  "lora_freq_hz": 433175000,
  "fox_freq_hz": 433920000
}
```

**Ограничения:**
- `station_key` — ровно 32 hex-символа (16 байт)
- `sector_key` — ровно 12 hex-символов (6 байт)
- `fox_morse` — не более 15 символов
- `cp_num` — 0–254 (255 зарезервирован как «не инициализирован»)

**Ответ 200:**
```json
{ "ok": true }
```

**Ответ 400:**
```json
{ "error": "bad body" }
```

**Ответ 500:**
```json
{ "error": "nvs save failed" }
```

---

### `POST /api/sync_time`

Установить системное время КП. Должен быть вызван до `POST /api/start`, чтобы отметки КП имели корректные временны́е метки.

**Тело запроса:**
```json
{ "timestamp": 1745000000 }
```

`timestamp` — Unix timestamp в секундах (UTC).

**Ответ 200:**
```json
{ "ok": true }
```

**Ответ 400:**
```json
{ "error": "missing timestamp" }
```

---

### `GET /api/status`

Получить текущий статус WiFi-портала. Используется для опроса состояния КП.

**Ответ 200:**
```json
{
  "portal_running": true,
  "config_valid": true,
  "cp_num": 5,
  "station_type": "regular",
  "client_connected": true,
  "uptime_sec": 47
}
```

| Поле | Описание |
|------|----------|
| `portal_running` | Всегда `true` пока сервер отвечает |
| `config_valid` | `true` если `initialized && cp_num != 255 && event_id != 0` |
| `client_connected` | `true` если хотя бы один клиент подключался к AP в текущей сессии (используется логикой таймаута) |
| `uptime_sec` | Время работы ESP32 с момента загрузки, секунды |

---

### `POST /api/start`

Команда запустить КП и завершить конфигурационную сессию. После успешного ответа WiFi будет отключён и КП перейдёт в рабочий режим.

> **Важно:** перед вызовом этого эндпоинта убедитесь, что конфиг записан (`POST /api/config`) и время синхронизировано (`POST /api/sync_time`). После отправки команды соединение с AP будет потеряно.

**Тело запроса:** пустое или `{}`

**Ответ 200:**
```json
{ "ok": true, "msg": "starting checkpoint" }
```

После этого ответа КП закрывает WiFi. Дальнейшие запросы на `192.168.4.1` не будут отвечать.

---

## 5. Порядок настройки (flow)

```
Пользователь подключается к AP "FoxAss-CP-XX"
              │
              ▼
      GET /api/status          ← проверить доступность и текущий config_valid
              │
              ▼
      GET /api/config          ← получить текущие настройки для отображения в форме
              │
              ▼ (пользователь заполнил форму)
     POST /api/config          ← записать новые настройки
              │
              ▼
    POST /api/sync_time        ← синхронизировать время (timestamp = Date.now()/1000)
              │
              ▼
      POST /api/start          ← запустить КП (WiFi отключится)
              │
              ▼
       Соединение потеряно ✓
```

**Таймаут WiFi:**
- Если никто не подключился в течение 10 минут после включения КП — WiFi отключается автоматически (КП переходит в рабочий режим)
- Если клиент подключился — таймаут отменяется автоматически, WiFi работает до явного `POST /api/start`
- КП без конфигурации (`initialized == false`) — таймаут не применяется, WiFi работает бессрочно до первой настройки

## 6. Криптография (для справки)

`master_secret` **никогда не передаётся** на КП и не хранится в его памяти. Веб-приложение (у организатора есть `master_secret`) вычисляет производные ключи и передаёт только их:

```
station_key = HMAC-SHA256(master_secret, cp_num)[0:16]
sector_key  = HMAC-SHA256(master_secret, event_id_big_endian)[0:6]
```

**Почему так:**
- Утечка конфига одного КП компрометирует только отметки этого КП
- `master_secret` остаётся известен только организатору и судейскому ПО

**Для веб-приложения:**
- При настройке КП №5 вычислить `station_key = HMAC-SHA256(master_secret, [0x05])[0:16]`
- `sector_key` одинаковый для всех КП одного соревнования
- Финишный КП (cp_num=254, 0xFE) использует `station_key = HMAC-SHA256(master_secret, [0xFE])[0:16]`

**Подпись отметки на карте (только для справки, делается на КП):**
```
MAC = HMAC-SHA256(station_key, cp_num || ts_arrive_be8 || flags || reserve_be2 || uid)[0:4]
```

---

## 7. Расширяемость

Все поля в `POST /api/config` опциональны и обрабатываются независимо. Для добавления нового параметра конфигурации:

1. Добавить поле в `StationConfig` (`components/config/include/config/types.h`)
2. Добавить сохранение/загрузку в `ConfigManager` (`manager.cpp`)
3. Добавить сериализацию в `handleGetConfig` и десериализацию в `handlePostConfig` (`wifi_portal.cpp`)

Существующие клиенты, не знающие о новом поле, продолжают работать — неизвестные поля в POST просто игнорируются, в GET — возвращаются вместе с остальными.

---

## 8. Примеры curl

```bash
# Получить конфиг
curl http://192.168.4.1/api/config

# Настроить КП №5, соревнование 2025
# station_key = HMAC-SHA256(master_secret, 0x05)[0:16]
# sector_key  = HMAC-SHA256(master_secret, event_id_be)[0:6]
curl -X POST http://192.168.4.1/api/config \
  -H "Content-Type: application/json" \
  -d '{"cp_num":5,"event_id":2025,"station_key":"a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6","sector_key":"1a2b3c4d5e6f"}'

# Синхронизировать время
curl -X POST http://192.168.4.1/api/sync_time \
  -H "Content-Type: application/json" \
  -d "{\"timestamp\":$(date +%s)}"

# Запустить КП
curl -X POST http://192.168.4.1/api/start

# Статус
curl http://192.168.4.1/api/status
```
